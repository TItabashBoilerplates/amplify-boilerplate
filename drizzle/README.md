# Drizzle ORM Schema Management

TypeScript-first database schema management with Drizzle ORM. Migration responsibility is fully consolidated under Drizzle (Supabase GitHub-integration migrations are not used).

## Overview

This project uses **Drizzle ORM** for declarative schema management with the following key features:

- **TypeScript-based schema** definitions
- **Declarative RLS policies** co-located with table definitions
- **Realtime publications** for real-time database subscriptions
- **Automatic type inference** for type-safe database operations
- **Drizzle-owned migrations** (output to `drizzle/migrations/`, v3 folder format)
- **Programmatic custom SQL execution** (pgvector, triggers, functions, realtime) via pre/post-migration phases

## Tech Stack

- **ORM**: Drizzle ORM 0.45.2
- **Migration Toolkit**: drizzle-kit 0.31.10 (v3 folder format)
- **Schema Validation**: drizzle-zod 0.8.3 (re-exported via `@workspace/db-schema`)
- **Database**: PostgreSQL (via Supabase)
- **Driver**: postgres 3.4.7
- **Package Manager**: Bun
- **Code Quality**: Biome (linting & formatting)

## Schema Structure

### Tables

This project (boilerplate baseline) manages the following database tables:

1. **users** (RLS) - Application users (linked to `auth.users` via the `auth_hook` trigger; `account_name` is auto-generated by `handle_new_user()` if not supplied via OAuth/OTP `user_metadata`)

> ボイラープレートはこの 1 テーブルのみを起点として提供する。プロファイル / 課金 / チャット / ベクトル検索など、プロジェクト固有のドメインテーブルは含まれていない。追加する場合は `drizzle/schema/schema.ts` にテーブル定義と RLS ポリシーを co-locate して、`devenv tasks run app:migrate-dev` でマイグレーションを生成・適用する。

### RLS Policies

All Row Level Security policies are **co-located with table definitions** in `schema/schema.ts`.

#### Example: users RLS

```typescript
// Table definition
export const users = pgTable('users', {
  id: uuid('id').primaryKey(),
  displayName: text('display_name').notNull(),
  accountName: text('account_name').notNull().unique(),
  // ... other columns
}).enableRLS()

// RLS policies (immediately following table definition)
export const insertPolicyUsers = pgPolicy('insert_policy_users', {
  for: 'insert',
  to: 'supabase_auth_admin',
  withCheck: sql`true`,
}).link(users)

export const selectOwnUser = pgPolicy('select_own_user', {
  for: 'select',
  to: ['anon', 'authenticated'],
  using: sql`true`,
}).link(users)

export const editPolicyUsers = pgPolicy('edit_policy_users', {
  for: 'all',
  to: 'authenticated',
  using: sql`(SELECT auth.uid()) = id`,
  withCheck: sql`(SELECT auth.uid()) = id`,
}).link(users)
```

#### RLS Policy Parameters

| Parameter | Values | Description |
|-----------|--------|-------------|
| `for` | `'select'`, `'insert'`, `'update'`, `'delete'`, `'all'` | Operation type |
| `to` | Role name(s) (array or string) | Target role(s) |
| `using` | `sql` template | Condition for viewing/editing existing rows |
| `withCheck` | `sql` template | Condition for inserting/updating new rows |

#### Complex RLS Example

```typescript
// Messages: Only visible to users who are members of the chat room
export const selectPolicyMessages = pgPolicy('select_policy_messages', {
  for: 'select',
  to: 'authenticated',
  using: sql`
    EXISTS (
      SELECT 1
      FROM user_chats
      JOIN users ON user_chats.user_id = users.id
      WHERE user_chats.chat_room_id = messages.chat_room_id
      AND users.id = (SELECT auth.uid())
    )
  `,
}).link(messages)
```

## Type Inference

Drizzle ORM provides **automatic type inference** for type-safe database operations.

### Generated Types

All tables export both SELECT and INSERT types:

```typescript
import type { InferSelectModel, InferInsertModel } from 'drizzle-orm'
import { users, chatRooms, messages } from './schema'

// SELECT types (existing records)
export type User = InferSelectModel<typeof users>
export type ChatRoom = InferSelectModel<typeof chatRooms>
export type Message = InferSelectModel<typeof messages>

// INSERT types (new records)
export type NewUser = InferInsertModel<typeof users>
export type NewChatRoom = InferInsertModel<typeof chatRooms>
export type NewMessage = InferInsertModel<typeof messages>
```

### Usage Example

```typescript
import { users, type User, type NewUser } from '@/drizzle'

// Type-safe insert
const newUser: NewUser = {
  id: crypto.randomUUID(),
  displayName: 'John Doe',
  accountName: 'johndoe',
  // TypeScript will enforce all required fields
}

// Type-safe select
const allUsers: User[] = await db.select().from(users)
```

## Custom SQL Management (pre/post-migration)

Custom SQL（PostgreSQL extension・関数・トリガー・realtime publication）は **2 フェーズ** で管理し、`migrate.ts` (Bun スクリプト) が順次実行する。Drizzle 公式の migrator はテーブル定義の DDL のみ責務を持ち、それを挟む形で pre / post を実行する。

### Phase Layout

```
drizzle/config/
├── pre-migration/          # 適用順: drizzle-kit generate/migrate より前
│   └── 00_extensions.sql   # 例: CREATE EXTENSION IF NOT EXISTS vector;
└── post-migration/         # 適用順: drizzle-kit migrate の後
    └── 00_functions.sql    # 例: generate_cuid() / handle_new_user() / auth_hook trigger
```

各ディレクトリ内の `*.sql` はファイル名アルファベット順で実行される。

### Execution Flow

```
ローカル開発  : bun run migrate:pre → bun run generate → bun run migrate → bun run migrate:post
本番デプロイ  : bun run migrate:pre →                    bun run migrate → bun run migrate:post
```

devenv の task ラッパー:
- `devenv tasks run db:migrate-dev` (ローカル)
- `devenv tasks run app:migrate-dev` (ローカル + 型生成)
- `devenv tasks run -P production db:migrate-deploy` (本番)

### Adding Realtime Tables

`drizzle/config/post-migration/` 配下の SQL に追記する:

```sql
-- 新しいテーブルを supabase_realtime publication に追加
ALTER PUBLICATION supabase_realtime ADD TABLE public.your_table_name;
```

### Error Handling

`migrate.ts` は SQL 実行で失敗した時点で即座に終了（exit code 1）する。1 ファイルでも失敗すれば pipeline 全体が止まり、CI が誤って緑にならない設計。

## Configuration

### drizzle.config.ts

```typescript
export default defineConfig({

  out: './migrations',               // Migration output (Drizzle owns the path; v3 folder format)
  dialect: 'postgresql',             // PostgreSQL dialect
```


## Migration Management

### Migration Files

- **Location**: `drizzle/migrations/<timestamp>_<name>/migration.sql` (drizzle-kit v3 folder format)
- **Metadata**: `drizzle/migrations/meta/_journal.json` + `meta/<timestamp>_<name>.json`
- **Generation**: Automatic via `drizzle-kit generate`
- **Version Control**: Committed to Git

### Migration Workflow

1. **Development**: `devenv tasks run app:migrate-dev` generates and applies migration locally (with type regeneration)
2. **Review**: Check generated SQL in `drizzle/migrations/<ts>_<name>/migration.sql`
3. **Commit**: Add migration files to Git
4. **Deploy**: `devenv tasks run -P <profile> db:migrate-deploy` applies to remote environments

### Why not Supabase migrations?

Supabase の GitHub 連携によるマイグレーション自動適用は **意図的に使っていない**。理由は、Drizzle のテーブル定義 DDL の **前後** で extension 適用 / DB function / trigger 等の SQL を流す必要があり、Supabase 側のパイプラインではこれらを差し込めないため。

GitHub 連携自体は Edge Functions のデプロイ等で引き続き利用する。マイグレーションだけを Drizzle 側に寄せ、`drizzle/migrations/` は Supabase が拾う `supabase/migrations/` パスとは別物として管理する。

### Rollback Strategy

Drizzle doesn't support automatic rollback. To rollback:

1. Delete the problematic migration file
2. Create a new migration with inverse changes
3. Apply the new migration

## Realtime Publications

Supabase Realtime allows clients to subscribe to database changes in real-time. This project manages Realtime publications through the post-migration phase.

### Configuration Location

Realtime publications are managed in `drizzle/config/post-migration/`:

```sql
-- Enable realtime for specific tables
ALTER PUBLICATION supabase_realtime ADD TABLE public.messages;
ALTER PUBLICATION supabase_realtime ADD TABLE public.chat_rooms;
ALTER PUBLICATION supabase_realtime ADD TABLE public.user_chats;
```

### Adding Tables to Realtime

To enable realtime updates for a new table, append to a `*.sql` file in `drizzle/config/post-migration/`:

```sql
-- Add table to realtime publication
ALTER PUBLICATION supabase_realtime ADD TABLE public.your_table_name;
```

### Removing Tables from Realtime

To disable realtime updates for a table:

```sql
-- Remove table from realtime publication
ALTER PUBLICATION supabase_realtime DROP TABLE public.your_table_name;
```

### Important Notes

1. **Service Configuration**: The `[realtime]` section in `supabase/config.toml` only controls whether the Realtime service is enabled and its port settings.

2. **Table-Level Configuration**: Actual table publications (which tables support realtime) are managed here in Drizzle.

3. **RLS Integration**: Realtime respects Row Level Security policies. Clients will only receive updates for rows they have permission to see.

4. **Client Usage**: Once enabled, clients can subscribe to changes:

```typescript
// Frontend example
const channel = supabase
  .channel('messages')
  .on('postgres_changes',
    { event: '*', schema: 'public', table: 'messages' },
    (payload) => console.log('Change received!', payload)
  )
  .subscribe()
```

## Extension Patterns (Reference)

ボイラープレートには含まれていないが、このリポジトリのアーキテクチャでよくある拡張パターンを以下に示す。実際に使う場合は `drizzle/schema/` に該当テーブルを追加し、`drizzle/config/pre-migration/00_extensions.sql` に `CREATE EXTENSION IF NOT EXISTS vector;` を追加してから `devenv tasks run app:migrate-dev` を流す。

### pgvector (vector similarity search)

```typescript
import { customType, integer, pgTable, text, timestamp, uuid } from 'drizzle-orm/pg-core'
import { sql } from 'drizzle-orm'

const vector = customType<{ data: number[]; driverData: string }>({
  dataType() {
    return 'vector(1536)'
  },
  toDriver(value: number[]): string {
    return JSON.stringify(value)
  },
})

export const embeddings = pgTable('embeddings', {
  id: integer('id').primaryKey().generatedByDefaultAsIdentity(),
  userId: uuid('user_id')
    .notNull()
    .references(() => users.id, { onDelete: 'cascade' }),
  content: text('content').notNull(),
  embedding: vector('embedding', { dimensions: 1536 }).notNull(),
  createdAt: timestamp('created_at', { withTimezone: true, precision: 3 })
    .notNull()
    .defaultNow(),
})

// Similarity search (raw SQL)
const similar = await db.execute(sql`
  SELECT content FROM embeddings
  WHERE user_id = ${userId}
  ORDER BY embedding <-> ${queryEmbedding}::vector
  LIMIT 10
`)
```

## Best Practices

### Schema Organization

1. **Co-locate RLS policies** with table definitions
2. **Define database enums via `pgEnum`** alongside the tables that use them (extract to a `types.ts` only if shared by multiple files)
3. **Export types** using `InferSelectModel` and `InferInsertModel`
4. **Document complex policies** with comments

### RLS Policy Design

1. **Start restrictive** - Only allow what's necessary
2. **Use auth.uid()** for user-specific checks
3. **Test policies** before deploying to production
4. **Document JOIN-based policies** for clarity

### Migration Strategy

1. **Generate migrations locally** before deploying
2. **Review migration SQL** before committing
3. **Test migrations** on staging environment first
4. **Never edit migration files** after committing

### Type Safety

1. **Always use inferred types** instead of manual type definitions
2. **Import types from schema** rather than duplicating
3. **Leverage TypeScript strict mode** for compile-time safety

## Integration with Other Services

### Frontend Integration

Frontend では 2 種類の型情報が利用可能:

```bash
devenv tasks run model:frontend
```

| 出力先 | 生成元 | 用途 |
|---|---|---|
| `frontend/packages/types/schema.ts` | Supabase CLI（`supabase gen types`） | Supabase クライアントの型推論 |
| `frontend/packages/db-schema/src/schema/` | drizzle/schema/ のコピー（auto-generated） | Drizzle スキーマ + drizzle-zod による Form / API payload validation |

`@workspace/db-schema` パッケージから zod schema を import できる:

```typescript
import { usersInsertSchema, type UsersInsert } from '@workspace/db-schema/zod'
import { z } from 'zod'

// プロフィール更新 Form スキーマ (アカウント情報を明示的に変更する画面)
const accountUpdateSchema = usersInsertSchema.pick({
  displayName: true,
  accountName: true,
})

// オンボーディング (初回プロフィール登録)
// accountName は handle_new_user() トリガーが auto.users から自動付与するので入力不要 → omit
const onboardingSchema = usersInsertSchema
  .pick({ displayName: true })
  .extend({ agreedToTerms: z.literal(true) })
```

#### DB トリガーで埋まるカラムの扱い (重要)

`drizzle/config/post-migration/00_functions.sql` の `handle_new_user()` トリガーで自動付与されるカラム (例: `users.accountName`) は **DB スキーマ上 NOT NULL** だが、**アプリケーション層では入力不要**。drizzle-zod は DB 制約に忠実に従って required にするため、Form/API payload では `pick` か `omit` で必要なフィールドだけ取り出す:

| シナリオ | 推奨パターン |
|---|---|
| サインアップ / OAuth コールバック | accountName は送らない (Trigger 任せ) → schema には含めない |
| プロフィール初回入力 (Onboarding) | `usersInsertSchema.pick({ displayName: true })` |
| アカウント設定の変更 | `usersInsertSchema.pick({ displayName: true, accountName: true })` |
| 部分更新 (PATCH) | `usersUpdateSchema.omit({ id: true })` (PK は path 由来) |

DB スキーマ由来の制約（NOT NULL / UNIQUE / max length / Enum 値）は drizzle-zod が自動反映するため、drizzle スキーマ変更時に zod 側の手動追従が不要になる。`drizzle/schema/` を編集 → `devenv tasks run model:frontend` で zod schema が自動再生成される。

### Edge Functions Integration

Edge Functions can use **both** Supabase types and Drizzle schemas:

```bash
devenv tasks run model:functions
```

Generated at:
- `supabase/functions/shared/types/supabase/schema.ts` (Supabase types)
- `supabase/functions/shared/drizzle/` (Drizzle schemas copied)

Usage in Edge Functions:

```typescript
import type { InferSelectModel } from 'npm:drizzle-orm'
import { users } from '../shared/drizzle/index.ts'

type User = InferSelectModel<typeof users>
```

